WRF (Weather Research and Forecasting) Model version 4.7.1 is a state-of-the-art numerical weather prediction system developed and maintained by the National Center for Atmospheric Research (NCAR) and the University Corporation for Atmospheric Research (UCAR). It is open-source public domain software widely used for both operational forecasting and atmospheric research, from regional mesoscale simulations to global climate studies.

The repository focuses on the **Advanced Research WRF (ARW)** dynamical core — a fully compressible, non-hydrostatic Eulerian solver on an Arakawa C-grid with Runge-Kutta time integration. WRF is designed with a hierarchical software architecture that cleanly separates scientific physics code from the underlying parallel computing infrastructure.

### Major Subsystems

The codebase is organized into several distinct functional subsystems:

- **ARW Dynamics** (`dyn_em/`) — The core solver: advection, diffusion, acoustic time-splitting, boundary conditions, nesting, and initialization routines for both idealized and real-data cases.
- **Physics Parameterizations** (`phys/`) — A large library of interchangeable physics schemes covering microphysics, cumulus convection, planetary boundary layer, land surface, radiation (longwave and shortwave), and urban canopy models.
- **Chemistry** (`chem/`) — WRF-Chem modules for gas-phase chemistry, aerosol processes, emissions, and dry/wet deposition.
- **Data Assimilation — WRFDA** (`var/`) — 3D-Var, 4D-Var, hybrid EnVar, and ensemble data assimilation system for ingesting conventional observations, radiance data, radar, GPS, and more.
- **WRF-Hydro** (`hydro/`) — A terrestrial hydrology coupling architecture that links hydrological routing models to WRF, activated via the `WRF_HYDRO` environment variable at compile time.
- **WRFPLUS / TL-AD** (`wrftladj/`) — Tangent linear and adjoint models used by the 4D-Var data assimilation system.
- **Framework** (`frame/`) — Software infrastructure: domain management, MPI/OpenMP parallelism via RSL_LITE, I/O abstraction, nesting communication, and the namelist configuration system.
- **Registry** (`Registry/`) — A declarative data registry that drives code generation for model state fields, I/O streams, nesting, and configuration — a central design point of WRF's extensibility.

### Build Systems

WRF supports two parallel build paths:

**Legacy build** (original):
```bash
./configure          # interactive platform/compiler selection → configure.wrf
compile em_real      # builds wrf.exe, real.exe in main/ and links to test/em_real/
```

**CMake build** (recommended for new users):
```bash
./configure_new      # interactive wizard → _build/wrf_config.cmake
./compile_new -j 12  # compiles and installs to install/bin/
```

The CMake build auto-detects netCDF, MPI, and OpenMP from the environment and supports multiple concurrent build configurations using separate `-d`/`-i` directories. Clean-up is handled by `./cleanCMake.sh`.

### Key Configuration Points

WRF's runtime behavior is controlled entirely through `namelist.input` (in the `run/` directory). Key namelist groups include:

- `&time_control` — simulation start/end times, restart, output intervals and formats (netCDF, GRIB1/2, pnetCDF, HDF5)
- `&domains` — grid dimensions, nesting configuration, map projection
- `&physics` — selection of physics scheme options (one integer per process per domain)
- `&dynamics` — solver options, advection orders, diffusion, damping
- `&bdy_control` — lateral boundary conditions

### Parallelism and I/O

WRF supports **shared-memory (OpenMP)**, **distributed-memory (MPI)**, and **hybrid** parallelism. The RSL_LITE communication layer handles domain decomposition and ghost-cell halo exchanges. I/O is abstracted through a plugin architecture supporting netCDF, parallel netCDF (pnetCDF), HDF5, GRIB1, GRIB2, and ADIOS2. Asynchronous "quilting" I/O processes can be dedicated to output, keeping compute processes fully occupied during history writes.
